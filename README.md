# Интеграционный проектировщик v7.6

Интеграционный проектировщик — это детерминированная модель для проверки и проектирования интеграционных решений. Пользователь описывает бизнес-процесс как граф: какие системы участвуют, какие шаги выполняются, через какие каналы они взаимодействуют, где есть таймауты, повторы, компенсации, статусы и ограничения. Движок анализирует эту структуру и формирует понятный архитектурный разбор на русском языке.

Модель не использует LLM. Она работает как rule-engine: одинаковый ввод всегда даёт одинаковый результат. Это важно для повторяемого архитектурного ревью, проверки обязательных контролей и подготовки проектных артефактов.

## Что выдаёт модель

После анализа пользователь получает:

- итоговый вывод по готовности решения;
- проверку готовности к production;
- список недостающих вводных;
- обязательный архитектурный чек-лист;
- найденные риски и слабые места;
- рекомендуемые архитектурные паттерны;
- варианты решения: минимальный фикс, production-компромисс и целевая архитектура;
- сценарную основу для разработки: основной поток, альтернативы, ошибки, задачи и критерии приёмки;
- Mermaid-диаграммы процесса и последовательности взаимодействий;
- DDL для возможных служебных таблиц;
- Definition of Ready и Definition of Done;
- мониторинг и эксплуатационные метрики;
- черновик контракта события;
- чек-лист проверок и тестов;
- план безопасного внедрения.

В v7.6 расширен раздел UI «Справочник инвариантов». Теперь это не просто список проверок, а обучающий справочник для проектирования процессов. Для каждого инварианта показано:

- развёрнутое объяснение простыми словами;
- как выглядит правильное решение;
- вопросы для архитектурного ревью;
- когда его применять;
- на каком этапе процесса проектирования его проверять;
- что именно проверить;
- почему это важно;
- как закрыть инвариант;
- какие последствия будут при нарушении;
- эталонный кейс правильного применения;
- как проверить соблюдение на практике;
- пример типовой ошибки.

Справочник по-прежнему строится из того же `invariant_catalog.py`, который использует модель, а UI дополнительно раскрывает каждый пункт учебным слоем. Поэтому пользователь видит живой каталог реальных проверок, но в формате, понятном для аналитика, разработчика, QA и владельца процесса.

## Что изменено в v7.0

В v7.0 добавлен отдельный раздел UI «Справочник инвариантов». Он открывается по адресу `/invariants` и через ссылку на главной странице. Справочник использует тот же файл `invariant_catalog.py`, что и анализатор, поэтому пользователь видит не отдельную документацию, а живой каталог проверок, которые реально применяются в модели.

Для каждого инварианта показано:

- код и область проверки;
- понятное описание на русском языке;
- вопрос, который нужно задать перед разработкой;
- почему этот пункт важен;
- как закрыть пункт в проектировании, контракте, БД или тестах;
- пример типовой ошибки и правильного направления исправления.

В справочнике есть поиск и фильтр по области. Например, можно найти `operUid`, `requestId`, `retry`, `DLQ`, `eventVersion`, `ПДн`, `rollback` и быстро понять, какие проверки нужно не забыть при проектировании.

## Что изменено в v6.9

В v6.9 добавлен большой универсальный каталог архитектурных инвариантов. Это не набор частных кейсов, а системный слой проверок, который помогает не забывать мелкие, но опасные детали при проектировании интеграций. Каталог содержит более 100 инвариантов и применяется по контексту: простому REST-сценарию не навязываются все проверки Kafka, а event-driven, highload, DWH, финансовые, регуляторные и legacy-процессы получают расширенную матрицу.

Категории инвариантов:

- бизнес-цель, границы процесса и ответственность;
- идентификаторы, scope уникальности, business key, lookup key и idempotency key;
- API и event-контракты, версии, enum, время, размер payload;
- основной сценарий, альтернативы, ошибки, отмена, ручной разбор;
- статусы, terminal-state guard и зависшие промежуточные состояния;
- целостность данных, system of record, Outbox, Inbox, конкуренция, Saga;
- асинхронность, Kafka/queue, DLQ, replay, offsets, partitions, hot partitions;
- синхронные API, timeout, deadline, retry, circuit breaker, bulkhead;
- внешние зависимости, webhook, файлы, batch, status inquiry;
- производительность, highload, backpressure, индексы, рост таблиц;
- безопасность, ПДн, подписи, replay-атаки, секреты, retention;
- наблюдаемость, correlationId, business metrics, DLQ/retry/replay metrics;
- комплаенс, аудит, ledger, lineage, регуляторные исправления;
- DWH, витрины, CDC, watermark, backfill, freshness;
- внедрение, migration, rollback, dual-run, rolling deployment;
- тестирование: E2E, дубли, out-of-order, DLQ/replay, нагрузка, безопасность и rollback.

Отчёт теперь показывает, сколько инвариантов применимо к конкретному процессу, сколько блокируют выпуск, сколько требуют внимания и сколько ещё нужно уточнить. Для этого добавлен файл `invariant_catalog.py`; ядро `engine.py` подключает его в разделе «Матрица деталей, которые нельзя забыть».

Важно: абсолютно все возможные ошибки во всех системах перечислить невозможно. v6.9 решает задачу иначе: фиксирует наиболее универсальные классы production-инвариантов, из которых выводятся частные ошибки вроде `operUid` без `operationType`, `requestId` без `targetSystem`, webhook без подписи, event без envelope, retry без idempotency, replay по неполному ключу и другие похожие проблемы.

## Что изменено в v6.8

В v6.8 добавлен слой anti-forgetting radar — матрица деталей, которые нельзя забыть при проектировании. Теперь модель не только находит явные риски, но и системно задаёт проверочные вопросы по областям: идентичность и ключи, контракты, статусная модель, целостность данных, восстановление, порядок событий, безопасность, наблюдаемость и внедрение.

Главная идея: не добавлять бесконечно частные кейсы, а проверять архитектурные инварианты. Например, ошибка `operUid` без `operationType` теперь обобщена до класса ошибок «идентификатор без области уникальности»: если общий адаптер, докатчик или универсальный сервис ищет запись только по `requestId`, `operationId` или похожему идентификатору, но используется в разных направлениях, модель попросит явно определить scope и составной ключ.

Изменения:

- добавлен раздел «Матрица деталей, которые нельзя забыть» в UI и Markdown-отчёт;
- добавлена проверка инвариантов по ключам, контрактам, статусам, recovery, ordering, безопасности, observability и миграции;
- добавлено правило `generic_identifier_scope_ambiguity` для обобщённых ошибок одиночного идентификатора без scope;
- сценарий и чек-листы теперь подсказывают не только найденные дефекты, но и вопросы, которые нужно задать до разработки;
- добавлены регрессионные тесты и benchmark-кейс на общий адаптер с `requestId` без scope.

## Что изменено в v6.7

В v6.7 модель получила сценарный слой для подготовки основы разработки. Теперь отчёт не ограничивается рисками: он строит основной поток, альтернативные сценарии, ошибочные сценарии, список задач для постановки и критерии приёмки. Это помогает превращать архитектурный разбор в черновик use case, постановки и тест-плана.

Также добавлен новый класс проверок для сложных ошибок ключей поиска и идемпотентности. Типичный пример: универсальный докатчик использует один `operUid` для разных подопераций, отправляет запросы в систему А и систему Б, а поиск выполняется только по `operUid`. Модель теперь подсвечивает, что нужен составной ключ, например `operUid + operationType + targetSystem`, и проверяет этот пункт в чек-листе, паттернах, DDL и сценариях приёмки.

Изменения:

- добавлен раздел «Сценарная основа для разработки» в UI и Markdown-отчёт;
- основной сценарий строится пошагово по графу процесса;
- альтернативные сценарии формируются по структуре: async, retry, DLQ/replay, внешние отказы, reconciliation, одинаковый `operUid` для разных `operationType`;
- ошибочные сценарии строятся из critical/high классов рисков;
- добавлены задачи для переноса в разработку и критерии приёмки;
- добавлены поля описания ситуации, ключей поиска и проектных ограничений;
- добавлен шаблон «Универсальный докатчик»;
- добавлено правило `ambiguous_composite_business_key` для коллизии одиночного операционного идентификатора;
- добавлено правило `missing_key_scope_for_shared_dispatcher` для универсальных сервисов без области уникальности ключа;
- DDL может предлагать composite unique index для `operUid + operationType`;
- добавлены регрессионные тесты на сценарный слой и составной ключ.

## Что изменено в v6.6

В v6.6 исправлена главная проблема больших отчётов: однотипные находки больше не печатаются десятками отдельных карточек. Движок сохраняет все срабатывания правил, но пользовательский отчёт группирует их в классы рисков и показывает список затронутых мест.

Изменения:

- добавлена агрегация однотипных findings по rule, severity, причине и способу исправления;
- в отчёте теперь отображается «класс риска» и количество затронутых мест, а не повторяющиеся одинаковые блоки;
- в UI для сгруппированного риска можно раскрыть список затронутых шагов;
- скоринг стал считать базовый штраф за класс риска и ограниченный дополнительный штраф за количество затронутых мест;
- блоки «Варианты решения» и «План внедрения» больше не дублируют длинные списки одинаковых блокеров;
- тестовый чек-лист строится по сгруппированным critical/high классам рисков;
- добавлен регрессионный тест, который проверяет, что повторяющиеся риски действительно группируются.

## Чем v6 отличается от v5

В v5 логика была построена вокруг именованных кейсов и классификации по ключевым словам. Такой подход быстро становился неподдерживаемым: каждый новый сценарий требовал отдельной ветки кода.

В v6 модель работает иначе. Она анализирует структуру графа: глубину синхронной цепочки, критический путь по latency, бюджет таймаутов против SLA, dual write, retry без идемпотентности, внешние блокирующие вызовы, fan-in, fan-out, порядок событий, наличие Outbox, Inbox, DLQ, replay, correlationId и других production-контролей. Новая бизнес-ситуация обычно становится новой комбинацией уже известных структурных свойств.

## Запуск

```bash
python app.py            # http://127.0.0.1:8110/
python run_tests.py      # полный регрессионный набор тестов
python bench.py          # бенчмарк сложных production-кейсов
```

Переменные окружения:

- `HOST` — адрес сервера, по умолчанию `0.0.0.0`;
- `PORT` — порт сервера, по умолчанию `8110`;
- `MAX_POST_BYTES` — максимальный размер запроса, по умолчанию 2 МБ;
- `APP_DIR` — директория для SQLite-истории разборов, по умолчанию `.architect6`.

## Какие риски проверяет модель

Модель проверяет следующие классы проблем:

- слишком длинная синхронная цепочка;
- блокирующий вызов внешней системы;
- retry без идемпотентности;
- несколько писателей финансовой или основной сущности;
- распределённая запись без компенсаций;
- бюджет таймаутов больше заявленного SLA;
- блокирующие шаги без timeout;
- dual write без Outbox;
- webhook или callback без Inbox-дедупликации;
- файловый или batch-канал в быстром пути процесса;
- DWH или аналитика в операционном core-flow;
- регуляторный процесс без аудиторского следа;
- неописанный порядок событий и partition key;
- синхронный fan-out на несколько потребителей;
- нестабильные или rate-limited зависимости;
- пиковая нагрузка выше rate limit партнёра;
- retry без лимита попыток и DLQ;
- входящий webhook без проверки подписи;
- потоковый consumer без lag/backpressure-контролей;
- highload ingestion без контроля горячих партиций и late events;
- multi-tenant поток без изоляции нагрузки;
- чувствительные данные без retention и маскирования;
- CDC без контроля пропусков, delete-событий, replay и сверки;
- миграция или strangler без cutover и rollback;
- длинный async-процесс без статусной модели;
- SPOF на критическом пути;
- инверсия таймаутов между родителем и дочерним вызовом;
- каскадное усиление retry;
- read-your-writes после асинхронной записи;
- внешний блокирующий вызов внутри async-обработчика;
- распределённый процесс без correlationId;
- служебные таблицы без политики роста;
- событие без обязательной event envelope;
- важный async-процесс без reconciliation;
- распределённая интеграция без наблюдаемости;
- клиентский API без модели ошибок;
- поиск и дедупликация по слишком широкому операционному идентификатору;
- универсальный сервис без области уникальности operationId/operUid;
- критичная система без владельца.

## Как добавить новое правило

Новое правило добавляется одной функцией с декоратором. Остальной код менять не нужно.

```python
@rule('my_rule', 'Категория')
def r_my(model):
    if <условие над model['graph'] / model['steps'] / model['meta']>:
        return [F(
            'high',
            'Краткий вывод полноценным предложением.',
            'Где найден риск',
            'Почему это важно для процесса.',
            'Что нужно сделать, чтобы закрыть риск.'
        )]
    return []
```

## Правило для текстовки

Любой новый пользовательский текст должен быть написан полноценным русским предложением. Короткие технические термины допустимы только в таблицах, коде, названиях паттернов и общеупотребимых интеграционных терминах.

- По умолчанию раскрыты только первоочередные действия, production gates и критичные пропуски вводных. Большие разделы остаются доступными, но не перегружают первый экран.
- В редакторе процесса шаги отображаются карточнее: каждая строка показывает подписи полей и лучше читается на широком и мобильном экране.
- Добавлены контекстные подсказки у сущности, SLA, ключей поиска, ограничений, статусов и ключевых полей.
- В результатах появились быстрые ссылки «Открыть поле ключей», «Открыть поле статусов», «Открыть поле ключевых полей», чтобы пользователь понимал, где исправлять проблему во вводных.
- В справочник инвариантов добавлен блок «10 инвариантов, которые чаще всего ломают интеграции».

## v7.6 — расширенный справочник инвариантов

В справочник инвариантов добавлен обучающий слой для пользователя. Теперь каждая карточка инварианта показывает не только вопрос проверки, но и полный контекст применения:

- когда использовать инвариант;
- на каком этапе проектирования его проверять;
- что именно проверить;
- почему это важно;
- как закрыть инвариант;
- последствия нарушения;
- эталонный кейс правильного применения;
- как проверить соблюдение на практике;
- пример типовой ошибки.

Справочник остаётся связанным с `invariant_catalog.py`: карточки строятся на основе того же каталога, который используется моделью в anti-forgetting матрице.

## v7.6 — справочник инвариантов переписан проще

В v7.6 справочник инвариантов переписан в более простом учебном формате без потери технических деталей. Для каждого инварианта теперь явно показано:

- что это значит простыми словами;
- когда использовать;
- на каком этапе процесса проверять;
- как выглядит правильное решение;
- как применить по шагам;
- плохой и правильный вариант;
- вопросы для ревью;
- последствия нарушения;
- эталонный кейс;
- как проверить на практике;
- пример ошибки.

Дополнительно добавлены мини-расшифровки терминов: область уникальности, идемпотентность, DLQ, replay, Outbox, Inbox, correlationId, backpressure, cutover.
