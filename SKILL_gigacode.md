# SKILL: Архитектурное ревью интеграции через SmartAdvisor

## Когда применять
Пользователь описывает бизнес-процесс или интеграцию (системы, шаги, потоки
данных) и хочет проверить архитектуру / найти проблемы / найти дыры в требованиях.
Триггеры: «прогони ревью», «проверь интеграцию», «прогони smartadvisor»,
«проверь архитектуру процесса», «найди проблемы интеграции».

## Что делать (последовательность)
Ты — агент с доступом к терминалу. Выполни ШАГИ строго по порядку.

### Шаг 1. Возьми описание процесса от пользователя
Это свободный текст, например: «Клиент оформляет заказ через API, списываем
оплату через внешнюю нестабильную платёжку, сохраняем статус в базу, шлём
событие в Kafka».

### Шаг 2. Преврати описание в JSON-форму по схеме
Составь JSON строго по этой схеме (только то, что есть в описании; неизвестное —
разумный дефолт):

```json
{
  "meta": {
    "name": "название процесса",
    "entity": "главная сущность (Order, Payment, Booking...)",
    "money": "no | indirect | direct",
    "customer_visible": "yes | no",
    "regulatory": true,
    "sla_ms": 0,
    "ordering": "no | per_entity | global",
    "load_rps": 0,
    "peak_factor": 1
  },
  "systems": [
    {"name":"имя","role":"internal|external|broker|db|legacy|analytics",
     "stability":"stable|unstable|limited|unknown","criticality":"low|medium|high",
     "rate_limit_rps":0}
  ],
  "steps": [
    {"order":1,"name":"что делает шаг","system":"имя из systems",
     "channel":"rest|grpc|soap|graphql|kafka|rabbitmq|queue|event|db|file|batch",
     "blocking":true,"writes_entity":false,"retry":"none|auto|manual",
     "idempotency":"none|key|natural","timeout_ms":0,"depends_on":0,
     "compensation":""}
  ]
}
```

Правила распознавания:
- внешний API / платёжка → role "external"; очередь/Kafka → "broker";
  БД → "db"; DWH/аналитика → "analytics"; остальное → "internal".
- "нестабильн" → stability "unstable"; "ограничен" → "limited".
- шаг сохранения/создания/проводки сущности → writes_entity true.
- синхронный вызов, ждущий ответа → blocking true.
- depends_on по умолчанию = номер предыдущего шага (первый = 0).
- деньги/оплата в процессе → meta.money "direct".

### Шаг 3. Запусти агента с этой формой
Сохрани JSON во временный файл и выполни в терминале:

```bash
echo '<здесь JSON одной строкой>' | python sa_agent.py --file -
```

(агент в каталоге проекта SmartAdvisor; он принимает JSON из stdin и терпим к
обёртке. Можно также сохранить в файл form.json и `python sa_agent.py --file form.json`.)

Полезные флаги:
- `--gaps` — только дыры в требованиях (что не описано);
- `--details` — полный разбор каждой находки с причиной и фиксом.

### Шаг 4. Покажи пользователю вывод агента
Выведи результат аудита как есть: вердикт, оценку готовности, «с чего начать»,
сгруппированные находки, полноту требований. Не пересказывай и не сокращай —
анализ детерминированный, он точный.

Если пользователь спросит «почему» по конкретной находке или «что ещё не
описано» — перезапусти с `--details` или `--gaps` соответственно.

## Важно
- Анализ выполняет движок SmartAdvisor (детерминированный, без модели). Твоя
  роль — только превратить текст в форму и запустить агента. НЕ придумывай
  находки сам, НЕ оценивай архитектуру от себя — это делает движок.
- Если в описании мало данных, форма всё равно валидна; агент покажет низкую
  «полноту требований» и список недостающего — это нормально и полезно.
