Жёсткая проверка ranking v4.6.1

Проблема v4.6:
- слой SOAP Legacy Adapter мог стать главным вариантом в сложном E2E-процессе;
- аналогичные ошибки были возможны для DWH, webhook/callback, BFF/read model/cache, external adapter, async job, data sync, near real-time decision;
- причина: сортировка вариантов в первую очередь по score, а не по архитектурной роли варианта; частный слой мог получить 100 и обогнать core-flow.

Исправления:
1. Добавлена логика различения top-level/core-flow и layer-only вариантов.
2. Для сложного core-flow частные слои больше не могут быть top-1:
   - SOAP Legacy Adapter Integration;
   - Data Pipeline / DWH;
   - Webhook Intake + Inbox Processing;
   - Fast Read / Cached Read Model;
   - External API Adapter with Resilience;
   - Reference Data API + Versioned Cache;
   - Async Job / Heavy Processing Flow;
   - Data Synchronization / Source-of-Truth Sync;
   - Near Real-time Decision Flow;
   - BFF/API Composition with Partial Response.
3. Изменена сортировка: сначала архитектурный приоритет/роль, потом score.
4. SOAP/file/DWH/webhook/financial/multi-source/async/near-real-time получили контекстные правила приоритета.
5. Финансовая операция с webhook теперь выбирает Financial Operation State Machine, а сам webhook остаётся intake-слоем.
6. Чистый webhook/callback без E2E всё ещё выбирает Webhook Intake + Inbox Processing.
7. Чистый DWH всё ещё выбирает Data Pipeline / DWH.
8. Карточка 360/multi-source aggregation всё ещё выбирает BFF/API Composition, а не Saga.
9. Legacy file-only всё ещё выбирает Batch/File Integration.

Проверки:
- test_integration_architect.py: 18/18 passed
- test_sa_full_coverage.py: 16/16 passed
- test_rank_guard.py: 3/3 passed
- py_compile: passed

Итог:
Главная ошибка v4.6 исправлена системно, а не точечно только для SOAP legacy.
